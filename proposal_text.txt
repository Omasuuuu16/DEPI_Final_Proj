Problem Statement College students represent one of the most health-vulnerable demographic groups in modern society. Academic pressure, irregular sleep schedules, poor dietary habits, reduced physical activity, and elevated stress levels collectively contribute to a significant rise in preventable chronic health conditions among this population. Despite the wide availability of wearable devices and institutional health surveys, the data generated by these sources remains fragmented across disparate systems, making it difficult to extract timely and actionable health insights. This project addresses that gap by building a production-grade, end-to-end ELT (Extract, Load, Transform) data engineering pipeline that ingests longitudinal student health observations, loads them into a structured local data store, transforms them through modular, version-controlled SQL logic, and ultimately surfaces actionable insights through cloud-hosted analytics and interactive dashboards. The pipeline classifies students into three health risk categories — Fit, At-Risk, and Unhealthy — while also tracking how individual health conditions evolve over time. A key data challenge acknowledged upfront is a significant class imbalance in the target variable: 87.4% of all records fall into the At-Risk category, with only 4.9% classified as Fit and 7.6% as Unhealthy. This imbalance is addressed explicitly in the transformation and reporting strategy to prevent misleading analytical conclusions. 2. Project Objectives The main objectives of this project are: Design and implement a full ELT (Extract, Load, Transform) pipeline to process longitudinal student health observations, ingesting raw CSV source files through Apache Spark and staging them in DuckDB before cloud promotion. Extract and process raw CSV data using Apache Spark, leveraging its distributed processing capabilities for efficient large-scale file reading, schema enforcement, and initial data validation. Load validated records into DuckDB as the local analytical staging layer, providing a lightweight yet high-performance relational environment that integrates natively with dbt for modular transformation. Apply modular, version-controlled SQL transformation logic using dbt to clean, encode, and enrich the staged data, including PII-aware handling of identity columns such as National ID, First Name, Last Name, and Date of Birth. Engineer meaningful features from available columns to compensate for absent variables, including composite health scores, BMI category bands, sleep adequacy flags, age derivation from Date of Birth, and weekly per-student aggregations. Compute and publish dbt Metrics to provide reusable, governed business metrics across the pipeline, enabling consistent reporting and downstream consumption by Power BI. Promote transformed and validated data to SQL Server as the structured relational serving layer, maintaining clear separation between raw, transformed, and metric-layer schemas. Build interactive Power BI dashboards connected to SQL Server and Microsoft Fabric, providing stakeholders with visibility into student health distributions, risk trends, and cohort comparisons. Promote the final serving layer to Microsoft Fabric and Azure Cloud to enable scalable, cloud-hosted analytics accessible to institutional stakeholders. Orchestrate the entire pipeline using Apache Airflow with robust DAG design, task dependency management, retry logic, and monitoring. Containerize the pipeline using Docker, expose pipeline results via a REST API, and automate testing and deployment through GitHub Actions CI/CD pipelines. Evaluate pipeline quality through data completeness and consistency metrics, and measure performance through DAG runtime and transformation throughput benchmarks. 3. Dataset Description The dataset contains 50,000 student health observations collected from college students across a synthetic longitudinal window. Each record corresponds to a distinct observation for an individual student, identified by a National ID and supplemented with personally identifiable information including First Name, Last Name, and Date of Birth. The dataset is well-suited for tracking individual health trajectories over time, as well as for cross-sectional population-level analysis. Temporal coverage for each student is captured through Start Date, End Date, and an Is Current flag, enabling slowly changing dimension (SCD) patterns to track which observation represents the most recent health snapshot for a given individual. Age is derived programmatically from Date of Birth during the transformation stage rather than being stored as a raw source value. Methodology The project follows a structured ELT data engineering workflow consisting of seven stages, progressing from raw CSV extraction through distributed processing, local analytical staging, modular transformation, metrics computation, relational serving, interactive visualisation, and final cloud promotion: CSV File Extraction → Apache Spark → DuckDB → dbt Transformations → SQL Server → dbt Metrics → Power BI → Microsoft Fabric / Azure Cloud Stage 1 — CSV File Extraction The pipeline begins with the source CSV dataset, which holds all student health and lifestyle observations in a flat-file format. This stage is responsible for identifying the source file, validating its presence and structural integrity, and making it available for ingestion into the distributed processing layer. A lightweight pre-flight check confirms that the expected number of columns is present, that the file is non-empty, and that PII columns — national_id, first_name, last_name, and date_of_birth — are accounted for before any downstream processing begins. Once validated, the file is passed directly to Apache Spark for distributed ingestion. Stage 2 — Apache Spark (Distributed Processing) Apache Spark serves as the primary extraction and initial processing engine. Rather than reading the CSV sequentially, Spark ingests the file in parallel partitions, enforcing the declared schema at read time and rejecting malformed rows into a separate error log for audit review. This approach provides a scalable foundation that can accommodate significantly larger datasets without changes to the pipeline architecture. During this stage, Spark performs the following operations: column type casting and null detection across all 22 columns; row-count validation to confirm completeness; deduplication checks on the national_id field to identify duplicate student records before they propagate downstream; and partitioned output writing to a staging location consumable by DuckDB. Spark connects directly to DuckDB via the next stage by writing Parquet-format intermediary files, which DuckDB reads natively and efficiently. Stage 3 — DuckDB (Local Analytical Staging) DuckDB acts as the local analytical staging layer, receiving the Parquet output produced by Spark and making it available for modular SQL-based transformation. DuckDB was chosen for this role because of its in-process columnar execution engine, which provides high-performance analytical query processing without the overhead of a separate database server. It integrates natively with dbt through the dbt-duckdb adapter, allowing transformation models to run directly against the staged data. The Raw schema within DuckDB preserves the source records without modification, providing a reliable rollback point and ensuring full auditability. All 50,000 records are loaded across all 22 source columns before any transformation logic is applied. From this staging layer, dbt picks up the data and begins the transformation workflow. Stage 4 — dbt Transformations All transformation logic is implemented using dbt (Data Build Tool), enabling version-controlled, modular, and testable SQL transformations running against DuckDB. The transformation strategy addresses both data enrichment and PII governance: PII pseudonymisation: national_id is hashed using a consistent one-way function; first_name and last_name are replaced with pseudonymous identifiers; date_of_birth is retained only for age derivation and then excluded from all analytical layers exposed beyond the transformation boundary. Age derivation: The age column is calculated from date_of_birth using a date-difference function against the pipeline execution date, producing an integer age in years that replaces the source date field in all downstream models. SCD handling: The start_date, end_date, and is_current fields are used to implement a Type 2 Slowly Changing Dimension pattern, ensuring that only the current record per student is included in standard aggregations while historical snapshots remain queryable. Categorical encoding: Ordinal columns (stress_level, sleep_quality, physical_activity_level, smoking_alcohol) are integer-encoded (Low=1, Medium=2, High=3 pattern). Nominal columns (diet_type, gender) receive one-hot encoded indicator columns. BMI classification: Four-band classification applied — Underweight (&lt;18.5), Normal (18.5–24.9), Overweight (25.0–29.9), Obese (≥30.0). Sleep adequacy flag: Boolean column derived from sleep_duration; values ≥7.0 hours flagged as adequate, supporting the WHO-recommended threshold. Composite activity score: Weighted combination of step_count, exercise_duration, and calorie_expenditure normalised to a 0–100 scale, serving as a proxy for the absent physical_activity_level granularity. Hydration adequacy flag: water_intake ≥2.0 litres flagged as adequate per general health guidelines. Weekly per-student aggregations: Average sleep_duration, heart_rate, step_count, and exercise_duration computed per student per calendar week, enabling temporal trend analysis. Health trend indicator: Direction of change (improving / stable / declining) in composite activity score across consecutive weekly aggregations per student, compensating for the absent mental_health_status column. Class imbalance handling: Stratified views created per health_condition class, ensuring minority class (Fit, Unhealthy) records are surfaced explicitly in all downstream reporting. All dbt models are schema-tested for primary key uniqueness, non-null constraints, referential integrity, and accepted-value ranges before results are promoted to the SQL Server serving layer. Stage 5 — SQL Server (Relational Serving Layer) Once dbt transformations complete and all schema tests pass, the cleaned, enriched, and pseudonymised records are written to SQL Server as the structured relational serving layer. SQL Server hosts two schemas: a Transformed schema containing the full feature-engineered dataset, and a Metrics schema populated by dbt Metrics in the following stage. This layer serves as the authoritative source for Power BI dashboards and downstream cloud promotion. SQL Server was selected for its native compatibility with Power BI DirectQuery mode, its support for row-level security configurations required for protecting residual PII-adjacent fields, and its role as a familiar relational engine within the DEPI program stack. The separation between the DuckDB staging layer and the SQL Server serving layer ensures that transformation experimentation can occur without affecting the stability of the reporting environment. Stage 6 — dbt Metrics dbt Metrics are defined on top of the SQL Server Transformed schema to produce reusable, governed business metrics that can be consumed consistently across all downstream reporting surfaces. Metrics defined in this layer include average composite health score per cohort, weekly active student counts, health condition distribution ratios, mean BMI by demographic group, and sleep adequacy rate across observation windows. By centralising metric definitions within dbt rather than embedding them ad hoc in dashboard queries, the pipeline ensures that all consumers — Power BI, ad-hoc SQL queries, and API endpoints — operate from a single consistent definition of each metric. This governance layer eliminates the risk of conflicting calculations appearing across different reporting surfaces and simplifies future metric updates to a single source-of-truth change. Stage 7 — Power BI (Visualisation and Reporting) Power BI dashboards are connected to SQL Server and the dbt Metrics layer to deliver interactive, stakeholder-facing reporting. Dashboard pages include: Population Health Overview: Distribution of Fit / At-Risk / Unhealthy students with demographic breakdowns by gender, age group, and diet type. Temporal Health Trends: Weekly and monthly average metrics per student cohort, highlighting seasonal fluctuations in sleep, activity, and stress. Risk Factor Correlation Panel: Cross-analysis of categorical behavioural features against health_condition outcomes, exposing which lifestyle patterns most strongly associate with each risk category. Individual Student Health Trajectory: Time-series view per pseudonymised student record showing progression or regression across key health indicators over the observation window, using start_date and end_date as the temporal axis. Supporting exploratory visualisations are produced using Matplotlib and Seaborn during the data profiling and quality validation phases. Once dashboards are validated, the pipeline promotes the serving layer data to Microsoft Fabric and Azure Cloud. Stage 8 — Microsoft Fabric / Azure Cloud (Cloud Promotion) Transformed and validated data is promoted to Microsoft Fabric and Azure Cloud as the final layer of the pipeline, enabling scalable, cloud-hosted analytics accessible to institutional stakeholders. The cloud layer follows a three-tier medallion architecture within Azure Data Lake Storage Gen2: Bronze Layer: Raw CSV files archived for auditability and future reprocessing, with PII fields intact but access-controlled. Silver Layer: Cleaned, encoded, and pseudonymised records from the dbt transformation layer, stored in Parquet format for efficient querying. Gold Layer: Business-ready aggregated views, dbt Metrics outputs, and feature-engineered tables available for serving through Microsoft Fabric and Power BI Service. Microsoft Fabric Spark Pools are used for any large-scale distributed transformation steps required during the silver-to-gold promotion. This cloud layer ensures that the pipeline is not constrained by local infrastructure and can scale to accommodate growing institutional datasets. Orchestration — Apache Airflow Apache Airflow manages the entire pipeline lifecycle. A primary DAG is defined with the following task dependency chain: extract_csv — Read source file and validate row count, schema, and PII column presence. spark_ingest — Trigger Apache Spark job to read, validate, and partition the source CSV into DuckDB-ready Parquet files. duckdb_load — Load Spark output into DuckDB Raw schema, confirming row count parity with the source file. dbt_run — Execute dbt transformation models in dependency order against DuckDB, including PII pseudonymisation, age derivation, SCD handling, encoding, and feature engineering. dbt_test — Run dbt schema tests and data quality assertions, including uniqueness, not-null, referential integrity, and accepted-value range checks. sqlserver_load — Write validated, transformed records from DuckDB to SQL Server Transformed schema. dbt_metrics_run — Execute dbt Metrics definitions against the SQL Server Transformed schema and populate the Metrics schema. azure_sync — Promote data to Azure Data Lake Storage Gen2 Bronze, Silver, and Gold layers via Microsoft Fabric. validate_pipeline — Assert row count parity, null checks across critical fields, class distribution thresholds, and PII field absence from cloud-facing schemas. notify — Send completion or failure alerts via email / Slack integration. Each task is configured with retry logic (3 retries, 5-minute exponential backoff) and SLA monitoring. The pipeline runs on a scheduled batch cadence with full observability through Airflow&apos;s built-in UI. 5. Future Improvements While the current pipeline delivers a complete, production-grade ELT solution, several enhancements are identified as natural extensions that would further increase the system&apos;s maturity, scalability, and analytical value. The following improvements are recommended for future iterations of the project: Automated Data Ingestion: Replace manual CSV file placement with an event-driven ingestion trigger, such as an Azure Blob Storage event or an API-based data feed, eliminating the need for human intervention at the extraction stage. Real-Time Data Processing: Introduce a streaming layer using Apache Kafka or Azure Event Hubs to support near-real-time health observation ingestion, enabling the pipeline to detect and alert on emerging health trends without waiting for the next scheduled batch run. Incremental Loading: Extend the DuckDB staging and dbt transformation layers to support incremental model materialisation, processing only new or updated records on each pipeline run rather than reprocessing the full dataset, significantly reducing runtime and compute cost at scale. Advanced Data Quality Monitoring: Integrate a dedicated data quality framework such as Great Expectations or dbt&apos;s built-in anomaly detection to continuously monitor data completeness, distribution drift, and schema evolution, with automated alerting when thresholds are breached. Expanded dbt Testing: Extend the current dbt schema tests to include cross-model referential integrity checks, custom singular tests for domain-specific business rules (such as BMI range validity and age plausibility), and freshness assertions on temporal columns. Machine Learning Integration: Incorporate a machine learning pipeline using Python and scikit-learn or Azure Machine Learning to train and serve predictive health risk models directly on the feature-engineered SQL Server serving layer, enabling data-driven classification beyond the rule-based health_condition target variable. Predictive Analytics Dashboards: Extend the Power BI reporting layer to include forward-looking predictive views, surfacing model-generated risk scores alongside historical actuals and enabling stakeholders to anticipate at-risk student cohorts before conditions worsen. Automated Reporting: Configure scheduled Power BI report delivery and automated email distribution of key metric summaries to institutional stakeholders, reducing the need for manual dashboard access while maintaining regular visibility into population health trends. Cloud Scalability Improvements: Migrate the DuckDB local staging layer to a fully cloud-native equivalent — such as Microsoft Fabric Lakehouse — for deployments requiring horizontal scalability, and configure auto-scaling for the Azure Data Lake Storage and Microsoft Fabric serving infrastructure. Role-Based Access Control and Data Security: Implement row-level security in SQL Server and Power BI to restrict access to student cohort data based on institutional roles, and enforce column-level masking policies to ensure that residual PII-adjacent fields are inaccessible to analytical users. CI/CD for the Data Pipeline: Extend the current GitHub Actions CI/CD configuration to include automated dbt model compilation checks, schema test execution on pull request, integration testing against a DuckDB test fixture, and one-click rollback for failed pipeline deployments. Data Lineage and Governance: Integrate dbt&apos;s built-in lineage graph with a data catalogue tool such as DataHub or Microsoft Purview to provide end-to-end visibility into how each field — particularly PII and derived columns — flows from the source CSV through every transformation layer to the final reporting surface. Additional Power BI Dashboards: Develop supplementary reporting pages covering academic term comparisons, inter-cohort health benchmarking, dietary and exercise intervention effectiveness analysis, and administrative summaries designed for institutional health programme coordinators. Integration with External Data Sources: Enrich the student health dataset by connecting to complementary institutional data sources such as academic performance records, campus facility usage logs, or national health statistics, enabling multivariate analysis of the relationship between lifestyle, academic outcomes, and population-level health benchmarks. 9. Conclusion This project demonstrates how modern data engineering principles can be applied to a real-world student health domain to build a structured, production-grade ELT pipeline. By extracting raw CSV data through Apache Spark, staging and loading it into DuckDB, applying modular transformation and PII governance logic via dbt, promoting results to SQL Server and publishing governed metrics through the dbt Metrics layer, and finally surfacing insights through Power BI dashboards and Microsoft Fabric / Azure Cloud, the pipeline delivers a robust, maintainable, and scalable architecture that covers the full data lifecycle from raw ingestion to analytical consumption. The dataset&apos;s inherent challenges — including severe class imbalance, the presence of personally identifiable information requiring careful pseudonymisation, the temporal complexity introduced by slowly changing dimension fields, and the longitudinal multi-record-per-student structure — are addressed explicitly through thoughtful feature engineering, PII governance enforced at the transformation boundary, SCD-aware model design, stratified reporting views, and transparent data limitation documentation. These design decisions ensure that the insights produced by the pipeline are both technically sound and analytically honest. The resulting system — encompassing Apache Spark extraction, DuckDB staging, dbt transformations, dbt Metrics, Airflow orchestration, SQL Server serving, Power BI dashboards, Microsoft Fabric / Azure Cloud promotion, Docker deployment, a FastAPI REST layer, and GitHub Actions CI/CD — represents a comprehensive application of the complete data engineering toolkit covered by the DEPI Digital Egypt Pioneers curriculum, positioning Byte Busters to deliver a high-quality capstone project aligned with industry standards.
